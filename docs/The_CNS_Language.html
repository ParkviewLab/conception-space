The CNS Language · conception-space

A hand-authored description language for 3D conceptual graphs. File extension .cns. This document covers the language itself — syntax, semantics, and the rationale behind each design choice. For the broader project intent that drives these choices, see the northstar.

Status note. As of v0.7.4 every construct described here ships. commonality renders as orbiting satellites, with an optional shape= attribute selecting the satellite geometry. An earlier opt-in style=lines (v0.7.2) was removed in v0.7.3 — the lines were visually indistinguishable from edge tubes. The Status / Migration section at the bottom is authoritative.
The Keyword Space

Four top-level keywords.
That is the whole language.

Every .cns file is some arrangement of declarations in these four. Each one names a structural concept; together they cover the entire surface of what an author can say.

1

node

An individual 3D object — a sphere, cube, octahedron — at a defined position.

2

edge

A relationship between two nodes, with optional label, colour, and thickness.

3

cluster

A spatial container with its own origin. Children positioned relative to it. Hierarchical.

4

commonality

A flat, non-spatial grouping of nodes that share some property. Multi-membership.

1.

node

An individual 3D object placed at a defined position.

node <id> ( key=value  key=value  ... )
AttributeTypeDefaultMeaning
locus [x,y,z]required Position in 3D space (relative to parent cluster, or world if none).
label string id Human-readable label drawn near the node.
shape enum sphere sphere, cube, tetrahedron, octahedron, dodecahedron, icosahedron, cylinder.
color hex parent's #rrggbb fill colour for the mesh.
size float 1.0 Scale factor.
brightness float inherited Light emitted by the node.
file string none Path to an attached file (.md, .pdf, etc.).
 
node mars ( locus=[16,-13,6]  shape=dodecahedron  color=#cc4422  label="Mars"  file="mars_notes.md" )

Why hand-place the locus Placement is argument. When the author writes locus=[16,-13,6], they are making the claim "Mars belongs here." That claim is the author's reasoning about how Mars relates to everything else in the file. Auto-layout would erase the claim. The language refuses to provide one. (See northstar Axiom 1.)

2.

edge

A directed relationship between two nodes.

edge <from-id> -> <to-id> ( key=value  ... )
AttributeTypeDefaultMeaning
label stringnone Drawn near the edge midpoint.
color hex grey #rrggbb colour for the line.
thickness float 0.05 Tube radius for the edge geometry.
 
edge sun -> earth ( label="orbits"  color=#aeb014  thickness=0.08 )

Why edges carry as much reasoning as nodes Edge thickness, colour, and label are visual channels the author uses to encode the weight and kind of a relationship. A thick gold edge says something different than a thin grey one. The language deliberately does not auto-infer edges from proximity or content similarity — every edge is an explicit claim. (See northstar Axiom 5.)

3.

cluster

A spatial container with its own origin / reference frame. Hierarchical.

cluster <id> ( key=value  ... ) {
    # child nodes, edges, or nested clusters
}
AttributeTypeDefaultMeaning
locus [x,y,z]required The cluster's origin in its parent frame.
label stringid Drawn near the cluster's origin.
color hex grey Default colour for child nodes that don't override.
brightness float inherited Ambient brightness inside this cluster's region.
N int none Goldberg-sphere subdivision count for the optional tube frame.
stube_color hex none Tube-frame edge colour (if N is set).
tube_thickness float none Tube-frame edge thickness.
 
cluster jupiter_system ( locus=[20,4,-8]  color=#1a3a6a  N=5  stube_color=#44FFFF ) {
    node jupiter ( locus=[0,0,0]    color=#c8944a  label="Jupiter" )
    node io      ( locus=[-2,2,2]   color=#ffee44  size=0.5 )
    node europa  ( locus=[2,2,2]    shape=octahedron  size=0.5 )
}

Why a cluster sets a reference frame Real-world hierarchies have natural origins — Jupiter's moons are positioned relative to Jupiter, not relative to some abstract world point. Letting the cluster carry its origin means the author writes moon positions in Jupiter-relative coordinates, which is how they actually think about them. The syntax matches the reasoning.

Why single-parent (hierarchical) A node can only be relative to one origin at a time — the spatial frame is intrinsically tree-shaped. Multi-parent membership would be geometrically incoherent. For non-spatial multi-membership, use commonality. (See northstar Axiom 2.)

4.

commonality

A flat, non-spatial grouping of nodes that share some property. Multi-membership.

commonality <id> ( key=value  ... ) { member-id, member-id, ... }
AttributeTypeDefaultMeaning
label stringid Human-readable label.
color hex grey Colour for the per-host satellite marker.
shape enum sphere Geometry of the satellite marker (same set as node).
 
# file scope, alongside edges
commonality inhabited       ( color=#ff8844 )  { earth }
commonality ice_moons       ( color=#88ccff )  { europa, enceladus, callisto }
commonality rocky_planets   ( color=#a08070 )  { mercury, venus, earth, mars }

Why "commonality" It names the reason for grouping — a shared characteristic — not the thing. Other candidates name the container (a set, a group, a collection); this one names what makes the container cohere. Earth is inhabited; Mars is rocky. These are commonalities shared across members. The keyword matches the cognitive operation: extracting shared features across examples — what cog-sci calls concept formation.

Why multi-membership matters Earth belongs to inhabited AND rocky_planets simultaneously — these are different properties. The construct enforces multi-membership by being intrinsically about shared properties (a "commonality of one" is nonsensical). Visually, multi-membership stacks: a node in two commonalities gets two satellites on its orbital ring, one per commonality, each in the relevant colour.

·

locus=

A defined place in 3D space. The attribute that anchors every node and every cluster.

Both node and cluster carry a locus= attribute specifying their position. Same attribute, same meaning, both contexts: the defined place this thing sits in space.

The wink to method-of-loci The previous attribute name xyz= was generic and technical. locus= is semantic: it says "here is a defined place." And it is the Latin root of the method of loci — the ancient memory technique of anchoring information to specific spatial locations. The same cognitive trick conception-space mechanises at scale.

Putting locus= on the most-frequent attribute in the language places that cog-sci wink in every coordinate of every .cns file — quiet, pervasive, unmissable once you know.

A worked example

solar.cns in the shipping syntax (every construct below ships as of v0.7.4). Earth lives in earth_system (spatial) but also belongs to inhabited and rocky_planets (commonalities) — no conflict. 

# Solar system — conception-space example.
# Node locus values are relative to their parent cluster's locus.

cluster solar_system ( label="Solar System"  N=6  color=#CCCC33  stube_color=#CCCC33 ) {

    node sun     ( shape=sphere      locus=[0,0,0]   color=#ffdd00  size=5.0 )
    node mercury ( shape=tetrahedron locus=[4,2,-2]  color=#b5a090 )
    node venus   ( shape=cube        locus=[7,-1,4]  color=#e8c060 )

    cluster jupiter_system ( locus=[20,4,-8]  N=5  color=#1a3a6a ) {
        node jupiter  ( locus=[0,0,0]    color=#c8944a )
        node io       ( locus=[-2,2,2]   color=#ffee44  size=0.5 )
        node europa   ( locus=[2,2,2]    shape=octahedron  size=0.5 )
        node ganymede ( locus=[-2,-2,-2] shape=dodecahedron size=0.5 )
        node callisto ( locus=[2,-2,-2]  shape=icosahedron  size=0.5 )
    }

    cluster earth_system ( locus=[10,2,-5]  N=2 ) {
        node earth ( shape=octahedron  locus=[0,0,0]  color=#4488ee )
        node moon  ( shape=icosahedron locus=[1,3,2]  color=#aaaaaa )
    }

    cluster mars_system ( locus=[16,-13,6]  N=3 ) {
        node mars   ( shape=dodecahedron locus=[0,0,0]     color=#cc4422 )
        node phobos ( shape=tetrahedron  locus=[-2,2,1.5]  color=#887060 )
        node deimos ( shape=cube         locus=[2,-2,-1.5] color=#998878 )
    }
}

# Edges — orbital relationships.
edge sun -> mercury  ( label="orbits"  color=#aeb014 )
edge sun -> venus    ( label="orbits"  color=#aeb014 )
edge sun -> earth    ( label="orbits"  color=#aeb014 )
edge earth -> moon   ( label="orbits"  color=#336655 )

# Commonalities — non-spatial groupings by shared property.
commonality inhabited     ( color=#ff8844 )  { earth }
commonality rocky_planets ( color=#a08070 )  { mercury, venus, earth, mars }
commonality ice_moons     ( color=#88ccff )  { europa, ganymede, callisto }

The thought behind the naming

The keyword choices weren't arbitrary. Four decisions shaped the keyword space; each one is the answer to a real design question.

1

Drop the node_ prefix

node, node_group, edge → node, edge, cluster, commonality

The old language mixed prefixed and un-prefixed keywords for no real reason (edge wasn't node_edge). The rename to cluster was already going to break files; the right moment to rationalize the whole space. Single-word, un-prefixed throughout — four structural concepts sitting as peers.

The reader sees each keyword as a structural concept in the language, not as part of a prefix family.

2

cluster for the spatial container

node_group → cluster

The original node_group did two jobs — spatial reference frame (xyz origin) and visual association (coloured shell). We split it so a node could belong to many flat associations without breaking the single-origin constraint. The spatial half needed a name. Considered: neighborhood (warm but long), region (clinical), realm (mystical), locus (singular-feeling). cluster won — intrinsically collective in English, short, and the locus= attribute carries the reference-frame role explicitly so the keyword itself can stay simple.

The keyword doesn't have to spell out "this is the spatial one" — the attribute does.

3

commonality for the flat construct

(no construct existed) → commonality

The flat-membership construct needed a name. Considered: set (precise but cold), group (synonym with cluster), collection (implies curation), category (defaults to single-membership), cohort (academic), trait (can be unique to one). commonality won because it names the reason for grouping rather than the thing, intrinsically requires multi (a commonality of one is nonsensical), and lands squarely in concept-formation / category-theory in cog-sci.

The keyword matches the cognitive operation — abstracting across examples to find a shared property.

4

Move locus to an attribute

xyz=[1,2,3] → locus=[1,2,3]

The word locus is precious — root of method-of-loci, the ancient memory technique. We wanted it in the language. Considered putting it in keyword names (node_locus, cluster_locus) but both had problems. Realisation: locus is not a kind of thing but a property. A node has a locus. That's exactly what an attribute is for. Putting it on the most-frequent attribute places the cog-sci wink in every coordinate of every file — and frees the keyword space to be clean and single-token.

The same word means the same thing in both contexts (a node's locus is its position; a cluster's locus is its origin).

Cog-sci threads running through the design

What the language deliberately doesn't have

Anti-features. Their absence is part of the design — each one would weaken the author's argument-making.

  • Auto-layout Violates Axiom 1 (placement is argument). An algorithm placing nodes erases the author's claim.
  • Auto-edge-inference Violates Axiom 5 (edges carry as much reasoning as nodes). The author decides which relationships exist.
  • Inheritance / classes Wrong vocabulary for the domain. A .cns file isn't an object hierarchy; it's a spatial argument.
  • Macros / generation The whole point is hand-authoring as sense-making. Programmatic generation removes the reasoning.
  • Auto-snap on coordinates If the author types locus=[10.0327, ...], that exact value is what they meant. No silent rounding.
  • Default styling Beyond safe minimums (grey edges, default sphere shape), every visual choice is the author's.
  • A separate "edit mode" syntax Editing happens in the viewer; the .cns file is the canonical representation but not the editing surface.

Status / migration

The language is in a transition. This page presents the post-rename syntax as primary. Here is what's actually shipping vs what's in-flight.

Shipping today (v0.7.4)

  • node declarations (with locus=).
  • edge declarations.
  • cluster declarations (hierarchical, spatial; carries locus= for its reference-frame origin).
  • commonality declarations (flat, file-scope, multi-membership) rendered as orbiting satellites — one marker per commonality, evenly spaced 360°/N around the host, with an optional shape= selecting the satellite geometry. Geometry in commonality_visual_study.md.
  • Nested clusters.
  • Comments via #.
  • All listed attribute value types (number, string, hex colour, array, enum).
  • Phase 1 of the Keyword Rationalization + Flat Multi-membership work (v0.7.0): node_group → cluster and xyz= → locus= across parser, renderer, examples, generators, and docs. Northstar Axiom 2 updated in lockstep.
  • Phase 2 (v0.7.1): commonality keyword + satellites render.
  • v0.7.4: commonality shape= attribute — pick the satellite geometry per commonality (same set as node).

Removed

  • Phase 3 (style=lines constellation render) shipped in v0.7.2 and was removed in v0.7.3 — the lines were visually indistinguishable from edge tubes. Carryover items (bookmark × commonality, additional render styles) live in in-flight_ideas.md.
